# Land-Siting Integration Test Handoff

This note explains how to test the land-siting integration branch of PyPSA-GB on another Windows machine. It is intended for a colleague who wants to check that the land-derived technical-potential workflow, downstream `p_nom_max` tightening, future capacity candidates, and location-constraint reports run coherently.

## 1. Get the branch

Clone the fork that contains the branch, then check it out:

```powershell
git clone https://github.com/<your-username>/PyPSA-GB.git
cd PyPSA-GB
git checkout codex/land-siting-integration-share
```

If the repository has already been cloned:

```powershell
git fetch
git checkout codex/land-siting-integration-share
```

## 2. Create the environment

Create and activate the PyPSA-GB environment:

```powershell
conda env create -f envs/pypsa-gb.yaml
conda activate pypsa-gb
```

If the environment already exists, update it instead:

```powershell
conda env update -n pypsa-gb -f envs/pypsa-gb.yaml --prune
conda activate pypsa-gb
```

The full workflow solve requires the normal PyPSA-GB solver setup on the test machine. If the configured solver is Gurobi, the machine must have a working Gurobi installation and licence.

## 3. One-command test

From the repository root, run:

```powershell
.\scripts\dev\run_land_siting_integration_test.ps1
```

By default, the script uses:

- scenario: `HT35_zonal_constrained`
- mode: `full`
- cores: `8`
- conda environment name: `pypsa-gb`

The default full mode validates the config, runs targeted land-siting unit tests, ensures `resources/land/technical_potential_Zonal.csv` is available, solves the constrained network target, and requests the location-constraint report.

## 4. Safer diagnostic modes

If the full run is too heavy or the solver/data setup is not ready yet, use one of these modes first:

```powershell
.\scripts\dev\run_land_siting_integration_test.ps1 -Mode dry-run
.\scripts\dev\run_land_siting_integration_test.ps1 -Mode smoke
```

`dry-run` validates configuration and checks the Snakemake DAG for the constrained solved target and location report. `smoke` also runs the targeted land-siting unit tests.

To change scenario or cores:

```powershell
.\scripts\dev\run_land_siting_integration_test.ps1 -Scenario HT35_zonal_constrained -Cores 4
```

To use a specific conda environment path instead of `conda run -n pypsa-gb`:

```powershell
.\scripts\dev\run_land_siting_integration_test.ps1 -EnvPath C:\Users\<user>\.conda\envs\pypsa-gb
```

## 5. Technical-potential data

The downstream policy integration needs:

```text
resources/land/technical_potential_Zonal.csv
```

This is normally generated by the land pipeline from raw GIS data. For colleague testing, the script can use a small bundled example file:

```text
data/land/examples/technical_potential_Zonal.csv
```

If the generated `resources/land/technical_potential_Zonal.csv` file is missing, the script copies the bundled example by default. This tests downstream integration and reporting, but it is not a replacement for rebuilding the full land-potential pipeline from raw data.

To force the raw land-pipeline rebuild instead:

```powershell
.\scripts\dev\run_land_siting_integration_test.ps1 -RebuildLandPotential
```

Only use `-RebuildLandPotential` when the raw land datasets are available locally.

## 6. Expected outputs

A successful full test should create or update:

```text
resources/network/HT35_zonal_constrained_solved.nc
resources/analysis/HT35_zonal_constrained_location_constraint_report.csv
resources/generators/HT35_zonal_constrained_renewable_candidate_oversubscription.csv
resources/generators/HT35_zonal_constrained_future_capacity_oversubscription.csv
logs/policy/apply_technical_potential_constraints_HT35_zonal_constrained.log
logs/analysis/generate_location_constraint_report_HT35_zonal_constrained.log
```

The generator oversubscription reports are pre-policy candidate-envelope diagnostics. The location-constraint report is the post-policy, solved-aware report and is the best place to check what changed in final network `p_nom_max` values.

## 7. What to check

After the run, inspect:

```powershell
Get-Content resources\analysis\HT35_zonal_constrained_location_constraint_report.csv -TotalCount 10
```

Useful columns include:

- `pre_policy_p_nom_max_mw`
- `post_policy_p_nom_max_mw`
- `land_binding`
- `existing_oversubscribed`
- `possible_preallocation_artifact`
- `constraint_outcome`
- `p_nom_opt_mw`

The intended interpretation is:

- upstream candidate scripts preserve FES allocation, land-cap metadata, and audit provenance
- downstream policy tightening is the authoritative stage for final network candidate caps
- existing fixed capacity is grandfathered
- FES future capacity remains an upper-bound envelope, not a forced equality target
